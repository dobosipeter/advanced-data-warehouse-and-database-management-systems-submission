\section{System Architecture}

The system is a real-time air quality monitoring and alerting platform ingesting data
from the OpenAQ public API into a PostgreSQL database, processing it through triggers
and transactions, and exposing it via a FastAPI REST backend and Streamlit GUI.

\section{Course Technologies Demonstrated}

The following topics from the Advanced DBMS course are actively demonstrated in the project:

\subsection{Loose Coupling --- Triggers + Log Tables (Chapter~2)}

\begin{itemize}[nosep]
  \item \textbf{Trigger \texttt{trg\_measurement\_generate\_pollution\_alert}} --- fires AFTER INSERT on \texttt{oltp.measurement\_raw}; matches the measurement value against active threshold rules and inserts an alert row into \texttt{oltp.pollution\_alert}.
  \item \textbf{Trigger \texttt{trg\_pollution\_alert\_enqueue\_event}} --- fires AFTER INSERT OR UPDATE OF status on \texttt{oltp.pollution\_alert}; enqueues a lifecycle event into \texttt{audit.pollution\_alert\_outbox} (the log/outbox table).
  \item This implements the course-prescribed \emph{loose-coupling pattern}: the measurement insert knows nothing about alerting; the alerting trigger knows nothing about downstream consumers. The outbox table decouples producers from consumers.
\end{itemize}

\subsection{Transactions \& Error Handling (Chapter~1 review, expanded)}

\begin{itemize}[nosep]
  \item Ingestion runs inside explicit transactions per location batch.
  \item Sensor-level \textbf{savepoints} allow partial rollback of invalid sensor data without losing valid data from the same location.
  \item Run status is tracked as \texttt{succeeded}, \texttt{partial}, or \texttt{failed}.
  \item Invalid measurements (null value, missing timestamp) are skipped and counted rather than rolling back the entire batch.
\end{itemize}

\subsection{Partitioning (Chapter~5)}

\begin{itemize}[nosep]
  \item \texttt{oltp.measurement\_raw} is \textbf{range-partitioned by month} on \texttt{measured\_at}.
  \item 25+ monthly partitions are created via a PL/pgSQL loop (Jan 2025 -- Feb 2027).
  \item Queries filtering by time benefit from \textbf{partition pruning}.
  \item Old partitions can be detached/archived without affecting live operations.
\end{itemize}

\subsection{Database Administration, Backups, Maintenance (Appendix~B)}

\begin{itemize}[nosep]
  \item \texttt{scripts/backup.sh}: automated \texttt{pg\_dump --format=custom} with configurable retention (default 7 days) and optional off-site upload.
  \item Scheduled via cron: \textbf{daily at 02:00 UTC} for full backup.
  \item VACUUM and ANALYZE are scheduled via \texttt{pg\_cron} or OS-level cron for table maintenance.
\end{itemize}

\subsection{Indexing (Appendix~A)}

\begin{itemize}[nosep]
  \item B-tree indexes on all foreign key columns and high-frequency query paths.
  \item Composite indexes: \texttt{(sensor\_id, measured\_at)} on measurement, \texttt{(parameter\_id, city, warning\_level)} on thresholds.
  \item Partial index on outbox: \texttt{WHERE processed\_at IS NULL} for efficient unprocessed-event retrieval.
\end{itemize}

\subsection{Automated Scheduling}

The pipeline is scheduled via system cron on the production VM:
\begin{itemize}[nosep]
  \item \textbf{Every 3 hours}: \texttt{run\_pipeline.sh ingest-etl} --- fetch new data from OpenAQ and load into DW.
  \item \textbf{Daily at 06:00}: \texttt{run\_pipeline.sh predict-only} --- generate next-period predictions.
  \item \textbf{Weekly Sunday 03:00}: \texttt{run\_pipeline.sh train-predict} --- retrain the ML model and predict.
  \item \textbf{Daily at 02:00}: \texttt{backup.sh} --- database backup with rotation.
\end{itemize}

\section{Functional Specification}

\begin{enumerate}[nosep]
  \item \textbf{Data Ingestion} --- Incremental and initial bulk load from OpenAQ v3 API.
  \item \textbf{Threshold Management} --- CRUD for per-city, per-pollutant alert rules (GUI + API).
  \item \textbf{Automatic Alert Generation} --- PostgreSQL trigger on measurement insert.
  \item \textbf{Outbox/Event Pattern} --- Trigger-based loose-coupling for alert lifecycle events.
  \item \textbf{Alert Review Workflow} --- GUI for transitioning alerts (open → reviewed → closed).
  \item \textbf{Transaction Safety} --- Explicit transactions with savepoints and partial-failure handling.
  \item \textbf{Range Partitioning} --- Monthly partitions on measurement table.
  \item \textbf{Indexing} --- B-tree, composite, and partial indexes on hot query paths.
  \item \textbf{Backup \& Maintenance} --- Automated pg\_dump with retention + scheduled VACUUM.
  \item \textbf{REST API} --- FastAPI with OpenAPI/Swagger auto-documentation.
  \item \textbf{Operational GUI} --- Streamlit pages: Station Explorer, Threshold Management, Alerts, System Status.
  \item \textbf{CI/CD} --- GitHub Actions deploys on push to \texttt{main} via SSH.
\end{enumerate}
